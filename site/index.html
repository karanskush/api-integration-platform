Entity Dependency DAG

Which endpoint produces which ID, where it flows, the full prerequisite chain
⌄

Why it matters: The #1 time killer in API integration is figuring out call ordering and where IDs come from. 60–80% of integration time is spent discovering undocumented third-party connections.

Plaid's 4-step token chain:

POST /link/token/create → link_token (expires 4h) → Frontend Link module → user authenticates → public_token (one-time-use) → POST /item/public_token/exchange → access_token (permanent) + item_id → POST /transactions/sync

Stripe Connect progressive unlocking:

POST /v1/accounts (minimum: country + type=custom) → account_id → PATCH /v1/accounts/{id} (add identity data) → charges_enabled capability unlocks → POST /v1/accounts/{id}/external_accounts → payouts_enabled unlocks

The DAG computes the "getting started" guide — no human needs to write it. When a consumer's agent asks "how do I make a transfer," it gets the whole prerequisite chain. The cross-team Slack thread never happens.

2

State Machine Map

Every state an entity can occupy, valid transitions, terminal states, irreversibility
⌄

Why it matters: Every serious API has entities with state machines that are poorly documented. Developers assume wrong transitions, hit unexpected errors, and spend days debugging.

Stripe PaymentIntent — 7 states, no failed state:

requires_payment_method → requires_confirmation → requires_action (3D Secure / redirect) → processing → succeeded (terminal) → requires_capture (auth-and-capture) → canceled (terminal) // Critical: NO failed state. Payment failure loops back // to requires_payment_method. Most developers assume wrong.

Stripe's old Sources API had TWO interdependent state machines (Source + Charge). Browser disconnection after payment could prevent Charge creation. Money would auto-refund. Developers had to track both IDs to prevent double-charging. PaymentIntents collapsed this into a single state machine — proving state machine complexity is a real, money-losing problem.

Plaid Item lifecycle: ITEM_LOGIN_REQUIRED fires when user changes password, MFA changes, or OAuth expires. This error never fires in sandbox — most teams never build the re-authentication path.

3

Failure Catalog

Every error mapped to its actual trigger, retryability, and concrete fix
⌄

Why it matters: Undocumented/unclear errors are a top-3 integration blocker. Generic messages like "400 Bad Request" provide zero diagnostic value. 52% of developers faced production crashes from unannounced breaking changes in 2024.

Stripe webhook signature — zero-clue silent failure: Framework parses req.body into a JS object. Re-serialized JSON doesn't match Stripe's original bytes. Signature verification fails. Error message gives zero indication of root cause. Fix: use req.rawBody.

AWS EventBridge — silent JSON trap: EventBridge Rules require PascalCase keys when targeting Batch. Developers use camelCase (natural from SDK examples). No error returned. Job runs with default configuration, silently ignoring all custom overrides.

Twilio Error 11200 — one code, five failures: HTTP Retrieval Failure covers: webhook URL unreachable, non-2xx response, expired SSL, 15-second timeout exceeded, AND server errors. All produce the same error code.

5

Webhook Behavioral Contract

Delivery timing, retry policy, ordering guarantees, known race conditions
⌄

Why it matters: Webhooks are where the most money gets lost in fintech integrations. Average failure rate: 3.5%. 15% of implementations have no retry mechanism. One SaaS company lost $14,000/month from failed Stripe webhook handling alone.

Stripe webhook race condition (~1% of signups):

0ms User completes payment 50ms Webhook handler starts processing 60ms User redirected to /checkout/return 70ms Eager sync checks DB: "processed?" → No (webhook uncommitted) 80ms Eager sync starts processing 90ms Webhook commits transaction 100ms Eager sync commits → DUPLICATE TRANSACTION

Three attempted fixes failed (DB locks → deadlocks, unique constraints → partial failures, Redis keys → stuck payments). Actual fix: single-writer pattern with a message queue.

Stripe's five silent webhook failure modes:

  1. Signature verification fails because framework auto-parses req.body
  2. Test-mode signing secret deployed to production
  3. At-least-once delivery causes duplicate processing
  4. 10-second timeout exceeded by synchronous heavy operations
  5. Stale data in webhook payload (event fired before latest update)

The universal dirty secret: Payment gateway docs bury this: "Set up a re-query service that polls for transaction status." This is an admission that webhooks alone are insufficient. You MUST implement polling reconciliation as a safety net.

Critical rule: Always generate idempotency keys client-side. If the server generates the key, a retry after timeout produces a new key, completely bypassing deduplication.
7

Cross-Provider Correlation Map

How IDs and entities flow across API boundaries in multi-provider workflows
⌄

Why it matters: Real integrations are never single-API. A payment flow might touch Stripe + Plaid + Onfido + SendGrid. Each uses different identifiers for the same logical entity.

Voice AI stack — zero automatic correlation: Twilio uses CallSid, ElevenLabs uses history_item_id, Vapi uses call_id. Teams check each provider independently and miss failures between them.

Shopify fulfillment across 3 APIs:

Shopify webhook (order_id, line_items) → Shippo POST /shipments/ (returns rate_object_id) → Shippo POST /transactions/ (returns tracking_number) → Resend POST /emails (sends tracking to customer)

ACH return codes: 85+ NACHA codes (R01–R85), each requiring different handling. Many providers collapse them into generic failures, hiding whether the error is retryable or terminal. Nacha compliance: unauthorized returns must stay below 0.5%; overall below 15%.

Probe Engine

Seven probe types. Safety-first.

The engine runs guided exploration across all seven layers, with strict safety enforcement for every probe type.

Schema Probe SAFE

Discovers request/response shapes, required vs. optional fields, value patterns, and type mismatches with the spec.

Entity Trace Probe WRITE

Traces ID flow between endpoints. Creates entities, captures all ID-shaped values, attempts to use them as input to every matching endpoint.

State Transition Probe WRITE

Creates entities, calls every relevant endpoint, attempts invalid transitions, maps terminal states and irreversible operations.

Error Trigger Probe SAFE

Systematically mutates requests — omits fields, wrong types, expired tokens, duplicate requests — to catalog every distinct error and its retryability.

Timing Probe SAFE

Measures webhook delivery latency, observes ordering across rapid event triggers, discovers timeout thresholds and retry policies.

Idempotency Probe WRITE

Sends identical requests with same key, tests key reuse with different params, discovers cache expiration and failure caching behavior.

Environment Diff Probe BOTH ENVS

Probes both sandbox and production (when keys provided), diffs response shapes, timing, error codes, and data patterns.

!

Read endpoints — probe freely. Write endpoints — only against dev/staging or explicit opt-in. Money-moving endpoints — require per-endpoint opt-in + budget. Rate limits — explicit consent on call budget. PII — strip/tokenize before storing. Every learned fact records which environment it came from.

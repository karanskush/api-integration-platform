01 The drift

Your docs say one thing.
Your API does another.

~75% of production APIs deviate from their own OpenAPI spec. Type drift, undocumented fields, endpoints that 404, webhook delays nobody mentions. TrueAPI catches it and tells the truth.

What the spec says

openapi.yaml 
# GET /accounts/{id}
responses:
  200:
    schema:
      id: string
      holder_id: string
      balance: integer
      status: enum [active, frozen]

The contract promises 4 fields. balance is an integer. status is one of active or frozen. Two states. Simple.

What the API actually returns

probed · 1,402× 
# observed after 1,402 probes against production
id: "acc_1A9z..."
holder_id: "usr_4F2..."
balance: "1245.50"      // integer // string!
status: "pending_review"  // undocumented state
risk_score: 87            // undocumented field
limits: { ... }            // undocumented nested object

4 drifts caught: wrong type on balance, two undocumented states (pending_review, locked), two undocumented fields. Your agent would hallucinate every one.

05 Real money-losing bugs

The failures nobody documents.
We catch them before you ship.

Every one of these has burned a real team. They're not in the spec, not in the sandbox, not in the docs — and your agent will hallucinate right past them.

$14K/mo
Stripe

Webhook race condition on signup

Eager sync check ran before the webhook DB commit. Three "fixes" (DB locks, unique constraints, Redis keys) made it worse. customer.subscription.created raced invoice.paid on 1% of signups.

FixSingle-writer pattern via message queue. TrueAPI's webhook contract flags this exact race in the explorer + warns the agent before the call.
launch-day
Plaid

ITEM_LOGIN_REQUIRED never fires in sandbox

Fires in production when a user changes their bank password, MFA, or OAuth expires. Sandbox doesn't simulate it. Teams ship without a re-auth path and discover it when real users churn.

FixDetected by TrueAPI's sandbox/prod divergence map. Build the Link "update mode" handler before launch — not after.
$85M
Synapse / Evolve

Batch reconciliation, not continuous

Ledger reconciliation in batches instead of continuous → drift between source-of-truth and partner state went unnoticed → $85M of customer funds frozen during the collapse.

FixWebhook contract + idempotency map make continuous reconciliation observable. Drift becomes visible the moment it happens.
"hours"
AWS EventBridge

camelCase silently ignored

EventBridge → Batch requires PascalCase keys (ContainerOverrides). camelCase from the SDK examples is silently dropped. Jobs ran with default config. No error. No warning.

FixCase-sensitivity rules in the failure catalog. The agent sees "this provider rejects camelCase silently" as a tool-call note.
double-charges
Stripe

Idempotency-Key caches failures too

A transient 500 gets "stuck" for 24h if you reuse the key. Same key + different params = 400. Most devs don't know this and rotate the key on retry, defeating idempotency.

FixIdempotency map encodes the 24h cache + failure caching behavior. Agent knows when to rotate keys vs. retry with the same one.
200+ orders
Stripe Checkout

checkout.session.completed ≠ paid

Team provisioned digital products on checkout.session.completed. That event fires before payment actually settles for async methods. 200+ orders shipped without payment collected.

FixState-machine map distinguishes "checkout completed" from "payment captured" — agent gates fulfillment on the right event.
